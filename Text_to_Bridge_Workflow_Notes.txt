Text to Bridge Workflow Notes
=============================

1. Project Purpose
------------------

Text to Bridge is a Python + Abaqus workflow system for bridge finite element model production and analysis automation.

The long-term goal is to start from bridge drawings, design notes, and structured text, then automatically produce reviewable Abaqus models, run analyses, diagnose errors, repair model inputs, extract results, and generate engineering reports.

The current implementation is deterministic and does not call external LLM APIs. It is structured so that future LLM agents can use each module as a tool.

2. Two Workflow Levels
----------------------

V1 focuses on analysis automation:

- read bridge JSON
- generate Abaqus `.inp`
- run Abaqus
- monitor `.log`, `.msg`, `.dat`, `.sta`, `.odb`
- parse errors and warnings
- classify common Abaqus issues
- apply deterministic repair rules
- rerun after repair
- extract results
- write JSON and Markdown reports

V2 focuses on model production:

- read a bridge semantic JSON file
- scan local Abaqus journal samples
- run deterministic model-production agents
- create a model plan
- write an Abaqus/CAE Python script
- optionally call Abaqus/CAE noGUI
- generate `.cae` and `.inp`
- optionally verify the generated model with Abaqus/Standard

3. Main Implementation Idea
---------------------------

The core design idea is to avoid direct conversion from drawings to Abaqus. Instead, Text to Bridge uses a semantic intermediate layer:

drawings / text / JSON
  -> Bridge Semantic Model
  -> Finite Element Model Plan
  -> Abaqus Builder
  -> Abaqus `.cae` / `.inp`

This makes the workflow auditable. Drawing or text recognition may be uncertain, but the semantic model and Abaqus build script can be inspected before analysis.

4. Multi-Agent Architecture
---------------------------

The V2 workflow contains these deterministic agents:

- DocumentAgent
- ReferenceAgent
- GeometryAgent
- IdealizationAgent
- MaterialAgent
- MeshAgent
- BoundaryAgent
- LoadAgent
- QaAgent
- AbaqusCaeScriptBuilder

Each agent owns one engineering responsibility. This makes the system easier to expand into a future multi-agent LLM workflow.

5. Reference Model Usage
------------------------

The local `samples` directory contains Abaqus `.cae` and `.jnl` reference models.

The system does not commit or upload these large files. It reads only local `.jnl` files during model production to detect modelling patterns such as:

- wire beam modelling
- solid extrusion modelling
- connector supports
- mesh seeding
- static steps
- gravity loads
- pressure loads

This allows the generated models to follow local Abaqus modelling habits without copying the sample models.

6. Beam Model Path
------------------

The beam path creates:

- a 3D wire main girder
- B31 beam elements
- a rectangular beam section
- material and density
- support sets
- displacement boundary conditions
- gravity
- line load converted from deck pressure
- static step

The beam path has been verified with Abaqus/Standard.

7. Solid Model Path
-------------------

The solid path was added to follow the style found in the samples:

- `BaseSolidExtrude`
- `HomogeneousSolidSection`
- `C3D8R`
- `seedPart`
- `generateMesh`
- support station partitioning
- bottom support node sets
- gravity
- deck pressure or equivalent top-node force fallback

The solid model has also been verified with Abaqus/Standard.

8. Key Parameters
-----------------

Important semantic input fields include:

- `project_name`
- `bridge_type`
- `analysis_type`
- `model_level`
- `spans_m`
- `deck.width_m`
- `girder.width_m`
- `girder.height_m`
- `materials`
- `supports`
- `mesh.target_size_m`
- `mesh.element_type`
- `loads`

For a solid model:

```json
{
  "model_level": "solid",
  "mesh": {
    "element_type": "C3D8R",
    "target_size_m": 2.0
  }
}
```

9. Main Commands
----------------

Run V1 dry analysis:

```powershell
python main.py --input bridge_fem_agent\examples\simple_girder_bridge.json --workdir runs\simple_girder_bridge --max-repairs 3 --dry-run
```

Run V2 beam model production:

```powershell
python main.py --workflow model-production --input bridge_fem_agent\examples\three_span_agent_bridge.json --workdir runs\three_span_agent_bridge_v2_cae --samples-dir samples --build-cae
```

Run V2 solid model production:

```powershell
python main.py --workflow model-production --input bridge_fem_agent\examples\three_span_solid_bridge.json --workdir runs\three_span_solid_bridge_v2_cae --samples-dir samples --build-cae
```

Run tests:

```powershell
python -m unittest discover bridge_fem_agent\tests
```

10. Innovation Points
---------------------

The main innovation points are:

- bridge semantic model as an auditable intermediate layer
- deterministic multi-agent model production
- reference-sample-aware Abaqus generation
- beam and solid model paths under one workflow
- generated Abaqus/CAE Python scripts for human review
- pre-generation QA reports
- rule-based analysis repair
- ODB extraction through Abaqus Python
- future LLM-agent compatibility without current LLM dependency

11. Current Limitations
-----------------------

The current system does not yet parse drawings directly. PDF/CAD/OCR parsing is planned for future `DrawingAgent` development.

The solid model is a simplified rectangular girder model. More realistic box girders, track slabs, rail systems, bearings, cables, arch ribs, and local detail models are future work.
